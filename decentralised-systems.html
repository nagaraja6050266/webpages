Chapter 01

The Centralized Model

A system where a single authority controls all data, processing, and decision-making. All requests flow through one central service — one source of truth, one point of control.

🏗️

Single Source of Truth

One database owns all data. Synchronous request flow: Client → Server → DB → Response. Tight coupling between all application layers.

✅

Strong Consistency

ACID transactions guarantee correctness. Easier transaction management, centralized logging, and monitoring. Vertical scaling by upgrading CPU/RAM.

⚡

Real-World Examples

Banking core systems, payment gateways like Stripe & Razorpay, SaaS platforms like Google Docs, and centralized exchange order-matching engines.

Stripe Razorpay Google Docs Banking Systems
Chapter 02

Where Centralization Breaks Down

Every centralized system carries inherent risks. When the single point of control fails, the entire system goes with it — and with it, user trust.

💥

Single Point of Failure

System outage means the entire service becomes unavailable. No fallback, no recovery path — just darkness.

📈

Scalability Bottleneck

The central node becomes the performance ceiling. Vertical scaling hits physical hardware limits eventually.

🔒

Trust Dependency

Users must fully trust the central authority with their data, privacy, and service continuity.

🚫

Censorship & Control

The authority can restrict, modify, or revoke access unilaterally. Users have no recourse.

🛡️

Security Risk

One breach can compromise the entire system. A single attack surface for all data and services.

Chapter 03

Enter: Decentralized Systems

A system where control, data, and computation are distributed across multiple independent nodes, instead of relying on a single central authority. Multiple nodes collaborate — each can handle requests, store data, and no single point controls the entire system.

🔄 No Single Point of Failure

If one node fails, others continue serving requests. The system degrades gracefully instead of collapsing entirely.

📊 Horizontal Scaling

Add more nodes instead of upgrading one server. Scale out, not up — virtually unlimited growth potential.

🌍 Reduced Latency

Serve users from geographically closer nodes. Data travels shorter distances, responses arrive faster.

🛡️ Improved Resilience

Failures are isolated, not system-wide. A node going down is a local event, not a global catastrophe.

Chapter 04

Core Mechanisms

How do decentralized systems actually work under the hood? Four fundamental mechanisms enable nodes to collaborate effectively without central coordination.

1

Replication

Same data stored across multiple nodes. Improves availability and fault tolerance. Leader → follower replicas keep data synchronized.

Leader-Follower Multi-Leader Leaderless
2

Partitioning (Sharding)

Data split across nodes — each handles a subset. Typically uses hash-based routing like user_id % N to determine which node stores data.

Hash Partitioning Range Partitioning
3

Distributed Coordination

Nodes need to agree on state. Tools like ZooKeeper and etcd handle leader election, configuration management, and distributed consensus.

ZooKeeper etcd Raft
4

Load Distribution

Requests spread across nodes using load balancers and service discovery. No single node bears all the traffic — work is shared evenly.

Round Robin Weighted Least Connections
Chapter 05

Communication Patterns

How do nodes talk to each other? Four distinct communication models, each with different trade-offs in coupling, latency, and complexity.

⚡ Synchronous

Direct request-response communication. Client waits for server reply. Simple but creates tight coupling.

REST APIs gRPC
✔ Simple to implement ⚠ Tight coupling

📨 Asynchronous

Messages and events instead of direct calls. Producers and consumers operate independently through message brokers.

Kafka RabbitMQ
✔ Loose coupling ✔ Better scalability ⚠ Harder debugging

🔗 Peer-to-Peer (P2P)

Nodes communicate directly with each other — no central coordinator needed. Every node is both client and server.

BitTorrent Blockchain Networks
✔ No central dependency ⚠ Discovery complexity

🌊 Gossip Protocols

Nodes randomly share state with other nodes. Information spreads epidemically — eventually all nodes converge to the same state.

Cassandra Distributed Clusters
✔ Highly resilient ⚠ Eventually consistent
Chapter 06

The CAP Trade-off

In distributed systems, network failures are inevitable. When a network partition occurs, the system must choose between consistency and availability — you can't have both.

Consistency + Partition Tolerance (CP)

Reject some requests to ensure all responses are 100% correct.

  • Strong correctness guarantee
  • Reduced availability during partitions
  • Banking transactions, distributed locks
ZooKeeper etcd Banking

Chapter 07

Conflict Resolution

When multiple nodes update the same data concurrently, conflicts are inevitable. How a system resolves these conflicts defines its reliability and user experience.

⏱️ Last Write Wins (LWW)

Latest timestamp wins. Simple but can lose data silently.

India writes "Alice" at 10:01, US writes "Alicia" at 10:02 → "Alicia" wins. "Alice" is lost.

Cassandra DynamoDB (optional)

🔢 Versioning / Vector Clocks

Each update carries version info. System detects conflicts instead of silently overwriting.

Node A → v1 → "Alice", Node B → v1 → "Alicia" → Conflict detected, both preserved.

Amazon Dynamo

🔀 Merge Strategies

Combine updates intelligently at the application level. Works when updates touch different fields.

One node updates phone, another updates address → merge both, no conflict.

Google Docs CRDTs

👤 Manual Resolution

System flags the conflict and a human resolves it. Most accurate but slower.

Git: two developers edit the same line → merge conflict → developer chooses.

Git SVN

Chapter 08

Distributed vs. Decentralized

These terms are often used interchangeably — but they are NOT the same. The key difference lies in control and ownership.

🔹 Distributed System

Computation/data spread across multiple nodes. Focus is on architecture and scaling. May still have central control and coordination.

Microservices Cloud Services (AWS) Distributed DBs

🔹 Decentralized System

No single controlling authority. Focus is on control and ownership. Nodes are independent, autonomous, and often untrusted.

Bitcoin Ethereum BitTorrent
Aspect Distributed Decentralized
Control Can be centrally controlled No central authority
Ownership Single organization Multiple participants
Coordination Often centralized (orchestrator) Distributed consensus
Trust Model Trusted nodes Trustless / semi-trusted

Chapter 10

When to Use What

Choosing the right architecture isn't about which is "best" — it's about which is right for your specific requirements. Use this decision framework.

1

Do you need strong, immediate consistency for every read?

✅ Yes

Use Centralized or CP Distributed. Banking, inventory, financial ledgers.

→ Not always

Eventual consistency may work. Continue to question 2.

2

Does a single organization own and control all nodes?

✅ Yes

Use Distributed. Microservices, cloud-native apps, CDNs.

❌ No

Multiple parties with no central trust? Continue to question 3.

3

Do participants need to operate without trusting each other?

✅ Yes

Use Decentralized. Blockchain, P2P networks, DAOs, open protocols.

→ Partial trust

Consider Federated or Hybrid models. Email (SMTP), Mastodon.

Chapter 11

Key Takeaways

The core insights to carry forward from this exploration of system architectures.

01

Centralization is simple, not safe

Easy to build but carries single-point-of-failure risk that can bring down everything at once.

02

Distributed ≠ Decentralized

Distributed is about architecture (scaling). Decentralized is about control (who owns the system).

03

CAP is a trade-off, not a choice

Network partitions will happen. You must decide: reject requests (CP) or allow stale reads (AP).

04

Conflicts are inevitable

In any system where multiple nodes write data, conflicts will occur. The resolution strategy matters more than prevention.

05

No architecture is universally "best"

The right choice depends on consistency needs, trust model, scale requirements, and organizational structure.

06

Eventual consistency is not broken

Many of the world's largest systems (DNS, social media, CDNs) run on eventual consistency — and work just fine.
