Introduction

Why Systems Evolve

Modern systems don't start complex — they become complex under pressure. As usage grows, engineers face unavoidable problems that force systems to evolve.

📈

Too Many Users

One machine can't handle all the traffic. Systems need to grow beyond a single server.

🌍

Users Spread Globally

Users across regions need data closer to them to reduce latency.

⏱️

Must Stay Available

Systems must stay responsive even during failures. Downtime is unacceptable.

🔐

Trust Cannot Be Central

Multiple updates happen concurrently. Trust cannot always be centralized.

Systems move from simple → scalable → resilient, not because we want complexity, but because we cannot avoid it.

Chapter 01

Centralized Systems (Starting Point)

Every system starts here. One server, one database, one source of truth. It's the default because it works — until it doesn't.

📋 In Our Example

User updates profile. Request hits one server. Database is updated. All reads return the same value.

Client → API Server → Database → Response
✅

Why This Works So Well

Single source of truth. No conflicting updates. Strong consistency (ACID). Easy debugging.

⚠️

Where It Breaks

As scale increases, the cracks appear: scalability limits, single point of failure, and global latency.

📈

Scalability Limit

One database handles all traffic. The central node becomes the performance ceiling.

💥

Single Point of Failure

If DB fails → system down. No fallback, no recovery path.

🌍

Global Latency

Users far from server experience delays. Geography becomes an enemy.

Centralized systems are simple — until they are not enough.
Chapter 02

Scaling Beyond a Single System

Instead of defining "distributed systems" formally, let's look at what engineers actually do first.

1

Add More Servers

Load is distributed across servers. Improves throughput.

Users → Load Balancer → Multiple Servers → Database
2

Move Data Closer to Users

Regional deployments and read replicas reduce latency for distributed users.

3

Split Data (Sharding)

Different users handled by different nodes. Data is partitioned across the system.

4

Remove Central Bottlenecks

Nodes become more independent. Coordination replaces control. This is advanced territory.

This evolution naturally leads us to distributed and eventually decentralized systems. We will formally distinguish them later.
Chapter 03

Replication & Partitioning

These are the two fundamental tools used during scaling.

P

Partitioning (Sharding) — Scaling the System

Splits data across nodes. Each node handles less data and the system scales horizontally.

user_id % N

Node A handles Indian users. Node B handles US users.

✔ Scales horizontally ⚠ Each shard becomes fragile
R

Replication — Making the System Reliable

Copies data across nodes. If Node A fails, Node B or C serves the request. Survive failures and improve availability.

User data stored in Node A, Node B, Node C. If Node A fails → Node B or C serves.

✔ Survives failures ✔ Improves availability

🏗️ Combined Architecture (What Real Systems Do)

Shard 1 → A, B, C | Shard 2 → D, E, F
Partitioning

Spreads load across the system

Replication

Protects data from failures

Partitioning divides the problem. Replication protects the solution.
Chapter 04

Distributed Coordination (Deep Practical Understanding)

This is one of the most misunderstood concepts, so let's ground it in reality.

🧠 The Core Problem

You now have multiple nodes: Node A (India), Node B (US), Node C (Europe). Each can accept requests, update data, and act independently.

❗

Problem 1: Conflicting Updates

User updates name: India → "Alice", US → "Alicia". A says "Alice", B says "Alicia". 👉 Who is correct?

❗

Problem 2: Duplicate Work

Background job: Clean inactive users. Without coordination: A runs job, B runs job, C runs job. 👉 Duplicate execution.

❗

Problem 3: State Disagreement

Node B crashes. A thinks B is alive. C thinks B is dead. 👉 Routing breaks.

✅ What Coordination Actually Does

Nodes agree on state

Everyone sees the same version of truth.

Nodes agree on decisions

One leader, one job runner, one source of authority.

Nodes avoid conflicts

No duplication, no split-brain, no chaos.

🔁 Real-World Flow: Profile Update + Background Job

Step 1

User Updates Profile

Node A receives update → "Alice"

Step 2

Node A Shares Update

A sends to B and C (gossip)

Step 3

Conflict Detected

B already has "Alicia" → conflict

Step 4

System Resolves

Last Write Wins or Merge logic

Step 5

Nodes Converge

All nodes agree on same value

Step 6

Background Job

Nodes vote → one becomes leader

C

🔐 Leader Election (Practical View)

Nodes say: "Who should handle coordination tasks?" Each node votes. Majority decides.

etcd → Raft (majority voting) ZooKeeper → leader election + locks
🗳️

Why Majority Matters

With 3 nodes, you need 2 votes to decide. This prevents two leaders at the same time (split-brain).

✅ Node A ✅ Node B ❌ Node C

→ Majority reached (2/3). Node A becomes leader.

Key Insight: Coordination is what allows independent nodes to behave like a single system. Without it, you don't have a system — you have chaos.
Chapter 05

Communication Patterns

How do nodes talk to each other? Four distinct communication models, each with different trade-offs in coupling, latency, and complexity.

⚡ Synchronous

Direct request-response. Simple, but tightly coupled.

Service A → Service B → Response
✔ Simple ⚠ Tightly coupled

📨 Asynchronous

Messages and events instead of direct calls. Producers and consumers operate independently.

Producer → Queue → Consumer

Profile update → event → email service processes later

✔ Decoupled ✔ Scalable

🔗 Peer-to-Peer

Nodes communicate directly with each other — no central coordinator needed.

✔ No central dependency ⚠ Discovery complexity

🌊 Gossip Protocol (Important)

Instead of sending updates to all nodes: node shares with a few, those nodes spread further. 👉 Like rumors spreading.

✔ Scales to large systems ✔ Handles failures naturally ✔ Eventually reaches all nodes

Chapter 07

Conflict Resolution

When multiple nodes update the same data: Node A → "Alice", Node B → "Alicia". How does the system decide?

⏱️ Last Write Wins

Simple. May lose data. The latest timestamp wins, older data is overwritten.

🔢 Versioning (Vector Clocks)

Detect conflicts precisely. Track causality between updates to identify true conflicts.

🔀 Merge

Combine intelligently. Works when updates touch different fields — no conflict when orthogonal.

👤 Manual

User resolves conflict. Most accurate but requires human intervention.

Conflict resolution is driven by business needs.

Chapter 08

Distributed vs. Decentralized (Clear Comparison)

Now that we've seen the evolution, let's define clearly. The key difference lies in control and ownership.

🔹 Distributed System

Work spread across nodes. Focus is on architecture and scaling. Control may still be centralized.

= We scaled the system

🔹 Decentralized System

Work AND control spread across nodes. Focus is on ownership and trust. No central authority.

= We removed central control

Aspect Distributed Decentralized
Work Spread across nodes Spread across nodes
Control Central authority No central authority
Ownership Single organization Multiple entities
Decision Making Centralized Distributed
All decentralized systems are distributed, but not all distributed systems are decentralized.

Chapter 10

Common Misconceptions

These are the most frequent misunderstandings engineers encounter. Getting these wrong can lead to poor architectural decisions.

❌ Myth

Distributed ≠ Decentralized

Distributed is about architecture (spreading work). Decentralized is about control (who owns the system).

❌ Myth

Sharding ≠ Availability

Sharding improves scalability, not availability. Without replication, each shard is still a single point of failure.

❌ Myth

Replication ≠ Consistency

Replication actually introduces inconsistency risk. Multiple copies means synchronization challenges.

❌ Myth

CAP ≠ Permanent Limitation

CAP trade-offs only matter during network partitions — not during normal operation.

❌ Myth

Eventual Consistency ≠ Incorrect Forever

Eventually consistent systems converge. DNS, social media, CDNs — all serve billions on eventual consistency.
